System Analysis Summary

Part A — Lab 1 & Lab 2 review: purpose, actors, functions, and pages

▼
PART A — Lab 1 & Lab 2 Review

📌 Main Purpose of the System

The Kafeteria Dinamik@JKE ordering system aims to replace manual paper-based ordering by providing a digital interface where customers can browse the menu, submit food orders, and receive confirmation — improving efficiency for cafeteria staff and a better experience for customers.

👥 Main Actors / Users (from Lab 2)

  • Customer — browses menu, places order, views confirmation
  • Cafeteria Staff — receives and processes orders
  • Admin — manages menu items and system settings

⚙️ Main Functions (Use Cases from Lab 2)

  • Browse Menu (view food items, prices, availability)
  • Place Order (submit customer name, items, quantity)
  • Select Payment Method
  • Receive Order Confirmation
  • Manage Menu (Admin)

📄 Pages Required for Frontend Prototype

  • Home — system introduction & navigation
  • Menu — display food & drink items with details
  • Order Form — customer order input form
  • Result / Information — order confirmation page

Target Users

Who is this system designed for?

▼
USER GROUPS FROM LAB 2 USE CASE DIAGRAM
👨‍🎓 Customer (Student / Lecturer)
👩‍🍳 Cafeteria Staff
🧑‍💼 Admin / System Manager

👨‍🎓 Customer

Students and lecturers who visit the cafeteria. They browse the menu, place food orders, select payment method, and receive order confirmation through the system interface.

👩‍🍳 Cafeteria Staff

Staff members who receive incoming orders and prepare food. They view submitted orders and update availability status of menu items when items run out.

🧑‍💼 Admin / System Manager

The administrator manages the menu list, user accounts, and system settings. The Admin role supports the overall function of the cafeteria system and ensures data accuracy. This frontend prototype supports all three user groups identified in the use case diagram from Lab 2. The interface is designed to be simple, readable, and self-explanatory for first-time users.
